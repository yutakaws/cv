# スキルシート　　名前:高山　裕

## 概要
- AWSを中心としたクラウド基盤の設計構築を経験し、AWSリソースのCloudFormationやTerraformによるIaC化にも取り組んできました。
- AIリソース(SageMaker,OpenSearch,Bedrock)やコンテナ(ECS)、サーバーレスリソースを利用したWebアプリのインフラ面の構築経験があります。
- DatadogやPagerDutyを活用したSRE的な仕組みづくりや自動化のPoC検証などにも携わり、顧客がアラーム監視の自動化において掲げていた課題を解決へと結びつけました。
- 個人のスキルアップだけでなく、チームに知見を還元することを常に意識しており、社内で知見のなかったコンテナについての勉強会を開き、社内の技術力の向上に貢献しました。

## 主な業務経歴

### 鉄道Webシステムの基盤設計と構築【AWS/Datadog】(2025年3月〜現在)

【プロジェクト概要】鉄道Webシステムのクラウドシフト案件に参画。マルチVPC構成のネットワーク設計、構築。監視ログ管理の要件定義、設計、構築。4名からなる詳細設計・構築チームのリーダーを担当。具体的には下記。

- IaC化(CloudFormation)<br>
システム追加の際に汎用的に展開できるよう、環境固有のパラメータを外だしをしたテンプレート構成となるよう設計。<br>
環境構築においてセキュリティ的に制約があったため、CloudFormationのスタック展開する運用方法や仕組みを確立。<br>
ネットワーク関連のAWSリソースをメインでコード化を担当(VPC,ALB,Certificate Manager,Transit Gateway)<br>
構築チームのコードレビュー。<br>
- Datadog 監視<br>
APIポーリング、メトリクスストリーム、Datadog Agent経由で監視経路を分けたメトリクス監視を設計。<br>
API経由のメトリクスにおいて、収集遅延が発生していたため、監視モニターに評価期間遅延設定を導入し課題を解決。<br>
各システムの監視ダッシュボードを、システムの異常だけでなく予兆や傾向を監視できるような構成で実装。<br>
Webページに対しての外形監視において、短時間のレスポンス遅延による誤検知が頻発していたため閾値設定を調整し、安定して実際の障害のみを検知できるよう改善。<br>
- 基本設計、詳細設計、テスト仕様書作成<br>
Datadog(外形監視、メトリクス監視、ログ管理)<br>
VPC単位でのシステム追加を行うためTransit Gatewayを利用したマルチVPCのネットワーク。<br>
- その他<br>
新人メンバーの教育・フォローアップ、WBS進捗管理、手順書/仕様書のレビューを実施。<br>
顧客へのヒアリング。(監視設計、ドメイン/ACM証明書設定)<br>

【発揮したバリュー】<br>
顧客からの要件をそのままDatadogの外形監視で閾値を厳格に設定した結果、検証テストで誤検知が多発していたため、遅延傾向を分析して顧客と基準をすり合わせ、閾値・リトライ条件を再設計しました。<br>
実障害のみを検知できるよう改善し、技術的な対応にとどまらず、相手と一緒に最適な解決策を作り上げる姿勢を大切にしました。<br>

### 自動車保険システムの基盤設計と構築【AWS】(2025年2月〜現在)

【プロジェクト概要】AIを利用した自動車保険Webシステム新規実装案件に参画。AI関連リソース利用した構成の耐障害設計、負荷分散、ログ・メトリクス監視、ログ管理の基本設計、詳細設計、構築。IAM詳細設計、構築。バックアップ/リストアの運用設計。サブリーダーを担当。具体的には下記。

- IaC化(Terraform)<br>
CI/CD(Github)にてTerraformファイル(AWSリソース)の構築・バージョン管理。<br>
パートナー会社から用意されたGithubリポジトリの利用手順や説明書が確立されていなかったため、不明確な部分を洗い出し手順を確立しチームメンバーへ連携し作業効率の向上。<br>
ECSのタスク定義で設定された環境変数について、アプリチームとの間で認識齟齬が多々発生し、コンテナデプロイが失敗することがあった。定期的なミーティングを設けて認識合わせを徹底した結果、作業品質を向上。<br>
- ログ管理・メトリクス監視<br>
アプリチームの要件で開発用の別AWSアカウントにECSログを、常時確認できるようにしたいという希望があったため、Cloudwatch Logs,Kinesis Data firehoseやS3レプリケーションルールによりログ連携経路を実装。<br>
ECSコンテナのオートスケーリングを常時接続数でスケールアウトする要件があり、対象のメトリクスでスケールアウト用のCloudwatch Alarmを構築し、ECSと連携することでオートスケーリングを実現。<br>
- 基本設計、詳細設計、テスト仕様書作成<br>
Act-Act構成を実現するためマルチAZによる耐障害・可用性。(ECSonEC2,Opensearch)<br>
マルチAZ構成のECSコンテナへの負荷分散。<br>
アプリチームの要件に合わせたログ・メトリクス監視。(Cloudwatch Alarm,Cloudwatch Logs,SNS,Kinesis Data firehose,S3)<br>
パートナー会社によって定められたセキュリティ要件に合わせたIAMポリシー・ロール。<br>
バックアップ/リストアの運用設計。<br>

【発揮したバリュー】
パートナー会社が定めたセキュリティの制約により、試験環境および本番環境のAWSコンソールに触ることができない事や、アプリチームとの認識齟齬が生まれたことによって、Terraformのapply実行による構築作業においてエラーが多々発生しトラブルにみまわれた。<br>
対策として定期的なミーティングを開催し正確に認識をあわせ、Terraformコード作成のルールやコードレビューをチームで徹底したため、納期に間に合い環境構築を実現できた。<br>

### 鉄道Webシステムの監視システム自動化PoC検証支援【AWS/Datadog/PagerDuty】(2024年7月〜2025年2月)

【プロジェクト概要】鉄道Webシステムの監視システム自動化PoC検証案件に参画。Datadog,PagerDutyを利用した監視システムの自動化の検証。サブリーダーを担当。具体的には下記。

- SRE<br>
課題としていた監視アラートのインシデント処理自動化をPagerDutyにて検証し実現。<br>
実際に想定されるアラートをDatadogの外形監視であるブラウザテスト上にて構築。<br>
PagerDuty RunbookAutomationにてShell Scriptを使ったジョブによって複雑なフローで対応する必要があるアラートの自動化を実現。<br>
プライベートサブネットへの外形監視を可能にするためにAWSでの検証環境構築。<br>
- その他<br>
テスト仕様書、報告書作成。<br>
構築手順書作成。<br>
顧客への検証報告。<br>

【発揮したバリュー】PagerDuty RunbookAutomationについては、公開情報がほとんどなく他のプラットフォームへの連携に課題がありましたが、率先して仕様調査と検証を重ねることで連携方法を確立し、顧客が課題としていた運用上の問題を解決に導きました。

### 自動車保険システムのセキュリティ通信PoC検証支援【AWS】(2024年4月〜2025年7月)

【プロジェクト概要】自動車保険システムのセキュリティ通信PoC検証案件に参画。マルチVPC構成のAWS Firewallを利用した通信検証。サブリーダーを担当。具体的には下記。

- セキュリティ検証<br>
AWS Network Firewallを利用した通信速度検証。<br>
AWS Network Firewallが別のVPCに構築する要件があったため、Transit Gatewayを利用したマルチVPCをAWSでの検証環境構築。<br>
- その他<br>
テスト仕様書作成。<br>
構築手順書作成。<br>

【発揮したバリュー】AWS Network Firewallの導入によって通信速度の遅延し原因が特定できなかったが、出力されたフローログを解析。原因としてAWS Network Firewallのステートルールの過多であることが判明したため、ルールの最適化を図ったことで速度の改善をすることができた。

- 基本設計、詳細設計、テスト仕様書作成、パラメータ設定、パラメータシート作成<br>
コールドバックアップ、ホットバックアップのEC2インスタンスが混在するため、正常に要件通りのバックアップを取るための機能。(Step Functions,Eventbridge,DyanamoDB)<br>

すぐリカバリに成功したため本番環境において大きなインシデントにはならず、またその対応力が結合テストにて発生したエラー対応においても活かされた。
